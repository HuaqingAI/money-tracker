# Product Brief Distillate: Money Tracker

## Requirements Hints

- **F1 账单导入的技术边界已调研确认：**
  - Android: NotificationListenerService 可捕获80-90%交易（支付宝/微信/银行推送）
  - iOS: 无法拦截通知，需Siri语音记账(3s) + Widget一键记账(1s) + 截屏OCR
  - 双平台通用：支付宝CSV(GBK编码) / 微信CSV(UTF-8) 手动导入
  - 解析规则需支持热更新（支付格式变更时无需客户端发版）
  - 架构：客户端只采集原始数据，所有解析/去重/分类在后端处理

- **金额存储必须用整数（分）：** 避免浮点精度问题，展示时 /100
- **日期时间UTC存储：** 客户端展示时转用户时区
- **AI识别结果需用户确认：** 不能自动入账，防AI幻觉/OCR误识别
- **账单健康度检测（F19）：** 对比已记录总额与实际支出总额，显示"记账完整度"百分比
- **异常检测（F21）：** 基于用户历史模式的实时纠错（如午餐300元 = 10倍异常）
- **推送节奏自适应（F27）：** 系统学习用户偏好，不是产品经理定的固定节奏
- **隐私边界（F20/F5）：** 共享账本中某些消费只有自己看得到，需权限控制

## Technical Context

- **中国落地版技术栈（已调研确认）：**
  - 前端：React Native (Expo) + Next.js 15 (App Router)
  - 跨端：Turborepo + Solito + Tamagui, 代码复用率~90%
  - 后端：Next.js Route Handlers 作为统一API层
  - 数据库：自建Supabase Docker (PostgreSQL 16 + GoTrue + PostgREST)
  - AI：DeepSeek V3.2（主力，成本仅GPT的1/35）+ DeepSeek R1（复杂推理）+ DeepSeek OCR 2（票据识别91%+）+ Qwen 3（降级备选）
  - 部署：阿里云ECS u1 (2核4G, 199元/年) + Docker Compose + Nginx + Let's Encrypt
  - 存储：阿里云OSS（S3兼容协议）
  - CI/CD：GitHub Actions -> 阿里云ACR -> ECS
  - 认证：Supabase GoTrue + 微信OAuth（需expo-wechat + 微信开放平台300元认证）

- **Supabase Cloud/Vercel/Claude API 在中国不可用：** 已有完整替代方案，API兼容，迁移成本低
- **ICP备案需提前启动：** Web服务必须备案，但MVP如果主要是API服务（移动App调用后端）影响可控
- **Android上架需覆盖国内商店：** MVP先覆盖华为+应用宝，后续扩展
- **Next.js后期可能需要拆分：** 大量定时任务/实时推送/大文件解析场景下Serverless不适合，届时拆独立worker（局部重构，非推倒重来）
- **MVP年度基础设施成本约2000-4000元**

## Detailed User Scenarios

- **大强（已婚家庭）完整场景：**
  - 触发：夫妻争论"这个月花了太多了吧" -> 打开APP有事实依据
  - 核心交互：共享账本各自记录自动汇总；消费标记"为谁花的"（自己/配偶/孩子/家庭共用）
  - 关键洞察需求："跟我们类似的家庭平均花多少？"（参照系）
  - 情感需求：减少家庭摩擦，把争吵变成讨论

- **小丽（月光白领）完整场景：**
  - 触发：月底发现钱没了（"月底震惊时刻"）
  - 核心障碍：忙的时候记不了，闲的时候想不起来（"遗忘黑洞"）
  - 痛感来源：单笔不贵（咖啡30元），但月度累计惊人（660元）（"拿铁效应"）
  - 关键需求：AI自动捕获解决遗忘问题 + "故事化财务报告"（不是冷冰冰的数字，是"你在XX上比上月多了30%"）

- **小美（副业人群）完整场景：**
  - 触发："我到底赚了多少" -> 需要月度/季度结算
  - 核心需求：一个人=多个财务实体（打工人+老板），需切换"财务身份"独立核算
  - 隐性成本陷阱：投流费记得住，时间/样品/售后赔偿记不住 -> 以为赚了其实亏了
  - 关键功能：副业利润计算器（收入-显性成本-隐性成本=真实利润）

- **人情往来完整场景：**
  - 触发：收到请帖 -> "该给多少？"
  - 数据结构：网络型（按"人"组织），非线性流水
  - 时间跨度：可能跨5-10年（结婚时的礼金，十年后还要参考）
  - 决策因子：历史金额 + 物价涨幅 + 关系亲疏变化 + 宴会规格
  - 衍生需求：人情支出季节性预警（"下月有3场婚礼，预计随礼约3000"）

## Competitive Intelligence

- **竞品定位：** 传统记账APP（随手记、鲨鱼记账等）、Excel、"不记账"
- **传统竞品的核心弱点：** 要么不能零记账（需手动输入），要么功能死板（没有AI助手体验）
- **人情账管理几乎无竞品覆盖** -- 蓝海市场
- **多身份核算无竞品覆盖** -- 现有APP假设用户只有一个财务身份
- **竞品AI能力调研未完成** -- 需确认随手记等是否已引入AI自动记账，以验证差异化是否真实存在

## Rejected Ideas / Out of Scope

- **企业/团队版：** 明确为未来规划，第一版不纳入
- **自建AI模型：** 早期完全没必要，直接调API
- **小程序作为主产品：** 推送能力受限，但可考虑作为验证渠道（未做最终决策，被搁置）
- **复杂后端框架（Java/Spring等）：** 一个人开发效率太低，不适合MVP阶段
- **Kubernetes/微服务：** MVP不需要，增加运维复杂度

## Open Questions

- **竞品AI能力调研（Research 2）未完成：** 需验证"AI零记账"差异化是否真实
- **F4-F13免费/付费分界线的具体功能划分：** 产品简报层面已确定原则（高成本+最有特色），PRD需细化每个功能的免费/付费边界
- **iOS零记账体验的用户接受度：** Android可实现80-90%自动捕获，iOS只能做到"极速记账"而非"零记账"，需考虑差异化的用户期望管理
- **正向激励设计（F24）：** 用成就感代替罪恶感的具体机制需在UX阶段设计
- **推送+对话双模式交互（F30）：** 日常不打开，推送掌握大局；临时需要开口问，AI秒回 -- 具体交互流程需在UX阶段细化

## Scope Signals

- **MVP 范围明确：** F1-F3 + 注册登录（Phase 1）
- **Phase 2 范围：** F4-F8（差异化功能，启动付费）
- **Phase 3 范围：** F9-F13（AI管家智能层）
- **消费情景模拟（F31）和风险压力测试（F32）：** 头脑风暴中产生，未列入核心功能表，属于Phase 3+的延展功能
- **人生大事财务规划（F33）：** 同上，远期功能
