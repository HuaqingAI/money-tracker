## 1. Epic 回顾

### 1.1 Epic 目标

Epic 0 的核心目标是搭建整个项目的基础设置，让后续开发工作可以聚焦在实际功能上，而不是反复处理项目骨架、数据库、质量门禁、部署和运维底座。

从现有产物看，Epic 0 已覆盖以下基础能力：

- Monorepo 与 workspace 骨架
- UI 设计系统基线
- Supabase schema / RLS / seed / 演化规划
- 代码规范、格式化、测试与质量门禁
- GitHub Actions CI/CD
- 生产部署编排与回滚脚本
- 监控、日志与健康检查
- worker 与后台调度基础设施

### 1.2 交付结果概览

基于 `sprint-status.yaml`，Epic 0 已被标记为 `done`，所有 Epic 0 story 也都已进入完成态；但 story 文档状态与 sprint 状态存在明显不一致：

- `0-4`、`0-8` 文档状态为 `done`
- `0-1`、`0-3`、`0-6`、`0-7` 文档状态仍为 `review`
- `0-5` 文档状态仍为 `in-progress`
- `0-2` 文档状态仍为 `ready-for-dev`

这说明技术交付已经基本完成，但流程收口和文档同步没有形成统一闭环。

### 1.3 成功之处

#### 系统性底座已经成型

Epic 0 不是单点功能，而是把开发、运行、部署、观测和后台执行整条底座打通。这个范围本身就有价值，因为后续 Epic 可以直接复用这些基础能力。

#### 团队具备深度排障能力

多个 story 都体现出团队可以追到根因，而不是停留在表层修补。例如：

- Expo / Metro / React Native 版本错配问题最终回到 SDK 对齐，而不是长期维持临时 patch
- 数据库 story 中大量 review patch 落在结构层，包括 RLS、约束、溢出风险、依赖顺序和健康依赖
- 部署与 worker story 能把已有日志、Sentry、healthcheck 复用起来，而不是重复建设

#### 质量意识在增强

后半段 story 的验证记录明显更完整：

- `0-6` 有 compose、deploy 脚本、dry-run、Docker build、API build、全仓测试等结构级验证
- `0-8` 有 lint、test、worker build、烟雾启动等针对性验证

这说明团队不是只关注“写完”，而是在逐步转向“可验证地完成”。

## 2. 问题与阻力

### 2.1 最大痛点：验证链路重、反馈慢

项目负责人明确指出，Epic 0 最大的问题是“慢”，尤其是基础设施验证复杂：

- Docker Compose 驱动的数据库与 Supabase 验证链路较重
- 本地环境问题容易放大验证成本
- 基础设施类 story 天然需要跨目录、跨服务、跨工具链验证

这个判断与 story 记录一致。Epic 0 的大量时间并不是耗在业务代码，而是耗在环境、编排、兼容性和验证。

### 2.2 本地开发环境稳定性不足

多个 story 反复暴露出环境相关问题，尤其是 Windows 本地开发：

- Expo / Hermes 本地构建问题
- Windows 长路径问题
- `next build` 在 Windows 本地的 symlink / `EPERM` 限制
- Docker、PowerShell、bash 之间的变量透传与执行差异

这些问题不一定都能在短期内彻底消除，但如果不明确“官方支持路径”，它们会持续干扰后续 Epic。

### 2.3 基础设施漂移风险真实存在

项目负责人最担心的问题是：后续几个 Epic 执行过程中，因为兼容性、版本或底座问题，被迫回头修改基础设施，甚至出现破坏性改动，影响已完成 story。

这是一个高质量风险判断，而且已有证据支持：

- `0-1` 出现过 post-merge 基础设施修复
- `0-2` 和 `0-4`/`0-5`/`0-6` 都记录了版本、Hermes、平台差异等遗留限制
- `0-7` 和 `0-8` 的能力建立在前置 story 之上，任何基础设施破坏都可能放大到多个后续 Epic

### 2.4 流程状态不可信

当前最明显的流程问题不是“没做完”，而是“做完后没有统一收口”：

- sprint 状态认为 Epic 0 已完成
- 多个 story 文档状态仍停留在 `review`、`in-progress`、`ready-for-dev`

如果这个问题不解决，Epic 1 开始后会产生两个后果：

- 计划层面对已完成内容的判断失真
- 后续 retro、质量门禁、发布准备的输入数据不可信

## 3. 关键经验

### 3.1 基础设施 story 必须把“收口”当成交付的一部分

Epic 0 的教训不是“基础设施难”，而是“基础设施只做实现不做收口，会持续拖慢后续开发”。  
收口至少包括：

- 文档状态同步
- 已知限制归档
- 标准开发环境声明
- 最小烟雾验证路径固定

### 3.2 版本管理必须以平台真值为准

在移动端栈里，手写版本号风险很高。Epic 0 已经证明：

- Expo 生态依赖应按 SDK 对齐
- React Native / Metro / Hermes / Expo Router 的组合必须看平台真值
- 临时 patch 可以止血，但不能替代版本治理

### 3.3 Review 在基础设施 story 中非常关键

`0-3`、`0-5`、`0-8` 的记录说明，review 不是锦上添花，而是基础设施 story 的核心组成部分。  
很多真正的风险点都不是初版实现发现的，而是 review 后暴露的。

### 3.4 “CI 可过”和“本地可用”需要明确边界

Epic 0 暴露的一个现实是：

- CI / Linux 可能是稳定的
- Windows 本地环境不一定等价

如果不明确官方支持边界，团队会不断把环境个例当成必须全部解决的问题，导致节奏被拖慢。

   - Owner：Sue（Project Lead）

2. **将基础设施变更与业务 story 解耦**
   - Owner：Alice（Product Owner）
   - 说明：若后续发现底座问题，需要单独建修复 story，而不是在业务 story 中倒挂修改

3. **把破坏性改动风险前置审查**
   - Owner：Charlie（Senior Dev）
   - 说明：凡涉及 SDK、依赖树、schema、部署编排、shared contract 的改动，先做影响面检查，再进入实现

## 7. 总结

Epic 0 的价值是明确的：它完成了整个项目的基础设置，让后续开发有机会真正聚焦在功能本身。  
Epic 0 的代价也明确：基础设施类工作验证重、反馈慢，而且暴露出本地环境和版本兼容性的持续风险。

本次回顾的核心结论不是“继续补基础设施”，而是：

- 接受 Epic 0 已完成
- 在 Epic 1 开始前做一次小范围但严格的基础设施收口
- 冻结已验证的基础设施组合
- 把环境限制、版本边界和状态同步纳入正式工程纪律

如果这一步做好，Epic 1 的执行重心就能真正转移到业务功能，而不是再次被底座问题牵着走。
