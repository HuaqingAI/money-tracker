## Overview

**Page Purpose:** "消费流水全景" -- 让小丽浏览本月所有消费流水，通过分类筛选发现隐藏的消费模式（"餐饮竟然花了这么多？"）。这是 L1+（看到消费真相）的列表级兑现：Dashboard 给的是概览，这里给的是每一笔。

**User Situation:** 小丽有两条典型进入路径：
- **从 Dashboard 分类卡片进入（场景路径）：** 在 Dashboard 看到"餐饮 ¥3,200"占比最高，好奇地点进去 -- "都是什么餐饮？"此时她带着明确的探索意图，期望看到餐饮分类下的具体消费明细。
- **从底部 Tab "交易" 进入（日常路径）：** 随手翻翻最近花了什么。没有明确目标，更像"刷流水"。这是仅次于 Dashboard 的最高频页面。

**Success Criteria:**
- 页面加载到可交互 <= 1s（感知性能）
- 分类筛选使用率 >= 30%（用户主动探索分类维度）
- 单次会话平均浏览交易条数 >= 10（用户在主动翻看而非一闪而过）
- 点击某笔交易进入详情的转化率 >= 15%

**Entry Points:**
- 底部 Tab "交易"（日常入口 — 无筛选参数）
- Dashboard 分类卡片点击（带 `category={id}` 筛选参数）
- 01.8 月度报表中分类卡片点击（带 `category={id}` + `month={YYYY-MM}` 参数）
- 03.2 Transaction Detail 返回
- 深度链接 / 推送通知点击

**Exit Points:**
- 点击某笔交易 -> 03.2 Transaction Detail
- 底部 Tab 切换到其他页面
- 点击 "+" FAB -> 06.1 Manual Entry
- 搜索结果中点击交易 -> 03.2 Transaction Detail
- 返回手势 / 返回键（仅从 Dashboard 分类卡片进入时，返回 Dashboard）

## User Mental State

### 从 Dashboard 分类卡片进入

| 维度 | 状态 |
|------|------|
| 触发事件 | 在 Dashboard 看到"餐饮 ¥3,200"，好奇点进来 |
| 情绪基调 | 好奇 + 轻微震惊（"餐饮怎么这么多？"） |
| 希望 | "让我看看都是什么餐饮消费，是外卖多还是堂食多" |
| 恐惧 | "别给我一堆乱七八糟的数字，看不懂就退了" |
| 内心问题 | "3200 都是什么？""奶茶到底花了多少？""有没有重复扣款？" |
| 耐心程度 | 中 -- 有明确探索目标，愿意翻看，但超过 30 条不分组就烦了 |
| 关键信号 | 列表中如果每天都有"瑞幸""蜜雪冰城"，她会忍不住数一下 |

### 从底部 Tab 进入（日常翻看）

| 维度 | 状态 |
|------|------|
| 触发事件 | 随手切到"交易"Tab，像刷朋友圈一样翻流水 |
| 情绪基调 | 平静 + 随意（"看看最近花了什么"） |
| 希望 | "快速浏览，有什么异常一眼看到" |
| 恐惧 | "别加载半天，别一片空白" |
| 内心问题 | "今天花了什么？""上周那笔扣款是什么？" |
| 耐心程度 | 低 -- 无明确目标，体验不好就切走 |
| 关键信号 | 最近的交易排最前面，时间分组清晰 |

## Content & Language Strategy

**核心叙事：消费流水全景**

交易列表是"真相镜子"的微观表达 -- 每一行是一面小镜子，映出一笔真实的消费。文案策略是**极简标签** -- 商户名 + 分类 + 金额 + 时间，四个要素就是一笔交易的全部真相。

| 维度 | 策略 |
|------|------|
| 金额展示 | 支出用中性色（不用红色），收入用绿色。不通过颜色制造焦虑 |
| 分类筛选文案 | 用分类名（"餐饮""交通"），不用"筛选""过滤"等技术词 |
| 搜索汇总 | "本月瑞幸 12 笔共 ¥216" -- 直接给出模式发现，L2+ 的微观兑现 |
| 空状态 | 温暖引导，不说"您还没有记录" -- 用"还没有交易记录"（更口语化） |

**语气：** 客观、简洁、中性。不评判，不建议，不催促。列表页只是"摆事实"。

**禁用词：** "超支""浪费""过度消费""你应该"（评判性语言）

**语言：** 界面简体中文，代码/后端英文

**边界情况：**
- **从 03.2 返回时保持状态：** 筛选条件、滚动位置、搜索状态均保留（不重新加载）
- **分类筛选后无交易：** 显示轻量空提示"本月没有{category}消费"，不显示引导卡片
- **月份切换到无数据月份：** 显示空状态（与当月空状态相同，但 CTA 按钮换为"查看其他月份"）
- **搜索跨月份：** MVP 搜索仅限当前展示月份。Phase 2 可扩展为全时间范围搜索
- **交易条目被用户修改分类后返回列表：** 如果当前有分类筛选且修改后的分类不再匹配，该条目从列表中移除（平滑动画）
- **同一秒内多笔交易：** 保持导入顺序（id 升序），不合并
- **金额为 0 的交易：** 正常展示（可能是红包、退款抵扣等）
- **单日交易数极多（50+）：** 正常展示，日期分组 header sticky 帮助定位
- **搜索结果跨多日：** 搜索结果仍按日期分组展示，但使用绝对日期（"4月10日"而非"今天"）
- **支出+收入混合月份：** 收入条目金额用绿色 "+¥" 前缀，与支出视觉区分
- **手动添加的交易 vs AI 捕获的交易：** 列表中不做来源区分（对用户透明），详情页展示来源

## Design Decisions

1. **支出金额不用红色** -- L2-（害怕数字罪恶感 10/15）明确告诉我们：红色 = 警报 = 审判。交易列表的支出金额使用中性文字色（text-primary），只有收入用绿色做正向区分。这不是"弱化"支出信息，而是"不放大"负面情绪
2. **日期分组而非纯时间线** -- 按天分组是"生活叙事"的自然单元。小丽想的是"今天花了什么""昨天花了什么"，不是"第 1 条""第 2 条"。每组 header 显示日期和当日小计
3. **分类 Chip 横向滚动** -- 分类可能有 8-10 个，固定区域放不下。横向滚动是移动端最自然的 Chip 交互模式，且不占用纵向空间。从 Dashboard 进入时对应 Chip 预激活，保持上下文连贯
4. **搜索结果汇总** -- "本月瑞幸 12 笔共 ¥216" 是 L2+（发现隐藏消费模式）的微观版本。不只是列出搜索结果，还立刻给出模式总结。这个小细节可能触发"截图发闺蜜群"的时刻
5. **从 03.2 返回保持状态** -- 小丽点进某笔交易看完详情后返回，列表必须保持之前的筛选条件和滚动位置。重新加载 = 打断浏览流，对耐心度低的用户是致命的
6. **AI 低置信度标记用小圆点** -- 不用问号、感叹号、黄色三角等"有问题"的视觉暗示。品牌色小圆点轻柔提示"你可以看看这个分类对不对"，不制造焦虑
7. **Summary Bar 在筛选时显示占比** -- "餐饮 ¥3,200 占总支出 24.9%" 比单纯的 ¥3,200 更有信息量。占比让小丽立刻理解"餐饮在我消费中的权重"，不需要她自己算
8. **左右滑动切换月份** -- 与 Dashboard 保持一致的手势交互，降低学习成本
9. **空状态引导到权限/导入** -- 交易列表的空状态不应该只是"没有数据"。引导用户去开启通知权限或导入账单，保持获取数据的路径畅通
10. **搜索 scope 包含金额** -- 小丽可能记得"上周有一笔 129 的扣款"，通过金额也能搜到。搜索范围：商户名、分类名、备注、金额

## Technical Notes

- **数据需求：** 每笔交易需要 id、商户名、分类/子分类、金额、时间、AI 置信度、数据来源字段
- **分页行为：** 每页约 20 条，滚动到底部自动加载更多。新交易插入不应导致用户看到重复条目或跳过条目
- **搜索需求：** 支持商户名、分类名、备注、金额的模糊匹配，前端 debounce 300ms
- **骨架屏：** 加载时展示 Summary 灰块 + 3 组日期 header + 各 3 条交易灰条
- **金额格式化：** 与 Dashboard 一致，千分位分隔符
- **滚动位置恢复：** 从详情页返回时保持之前的筛选条件和滚动位置（不重新加载）
- **筛选参数传递：** 从 Dashboard/报表进入时携带 category 和 month 参数，底部 Tab 进入不带参数
- **搜索历史：** 本地存储最近 5 条搜索词
