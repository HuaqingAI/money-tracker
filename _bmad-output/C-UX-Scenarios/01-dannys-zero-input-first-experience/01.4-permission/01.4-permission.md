## Overview

**Page Purpose:** 解释为什么需要通知读取权限（连接零输入承诺），用具体的隐私承诺消除数据安全恐惧（D5-），并提供厂商适配的设置引导帮助用户精准找到开关页面。

**User Situation:** 大强刚完成注册，期待正高。但系统即将问他要一个"高敏感"权限 -- 读取通知消息。他的银行扣款、支付宝消费、微信转账的通知都在这里，这些是他最私密的财务数据。D5- 在此刻被最大化触发："我刚注册就要读我通知？安全吗？"

**Success Criteria:**
- 授权转化率 >= 60%（首次进入本页即完成授权）
- 跳过后 7 天内回来授权的比例 >= 30%
- 零用户因"不知道怎么开"而放弃（厂商引导覆盖率 100%）
- 用户能说出"它不上传原始通知"这个关键承诺

**Entry Points:**
- 从 01.3 注册/登录成功（新用户）自动进入

**Exit Points:**
- 授权成功 -> 01.5 账单导入页
- 点击"暂时跳过" -> 01.7 Dashboard（空状态）
- iOS 用户 -> 01.5 账单导入页（通知权限不适用，引导 CSV）

## Content & Language Strategy

**叙事弧线（三段式信任建设）：**

| 段 | Section | 心理层 | 效果 |
|----|---------|--------|------|
| 第一段：为什么 | Hero | "原来要实现零记账就得读通知" | 将权限需求合理化（不是贪婪，是必要） |
| 第二段：怎么保护 | Privacy Commitments | "数据在手机上处理，不上传原文" | 正面回应 D5- 恐惧 |
| 第三段：怎么开+可以不开 | Guide + Skip | "手把手教我开，而且随时能关" | 消除操作障碍 + 给退路 |

**语气：** 诚恳、透明、不施压。像一个值得信任的朋友在解释"为什么需要你家钥匙"。

**关键信任词：** "你的手机上""绝不离开""只提取""加密""随时"

**禁用词：** "必须"、"需要您授权"（命令感）、"放心"（空洞）

**语言：** 界面简体中文，代码/后端英文

## On-Page Interactions

| 交互 | 触发 | 行为 | 反馈 |
|------|------|------|------|
| 页面加载 | 注册成功进入 | 检测设备平台(iOS/Android)和厂商 -> 显示对应内容 | 内容淡入 |
| 点击"去开启通知读取" | 触摸主按钮 (Android) | 打开系统通知使用权设置页 (Intent) | APP 进入后台 |
| 从系统设置返回 APP | 用户切回 APP | 检测 NotificationListenerService 是否已启用 | 见下方返回状态处理 |
| 权限已授权返回 | 检测到权限已开启 | 自动导航到 01.5 账单导入 | 成功动画（对勾 + "已开启"）+ 1s 后自动跳转 |
| 权限未授权返回 | 检测到权限未开启 | 保持当前页面，不做额外打扰 | 页面不变，用户可再次尝试或跳过 |
| 点击"暂时跳过" | 触摸跳过链接 (Android) | navigate to 01.7 Dashboard（空状态） | 页面淡出 |
| 点击"去导入账单" | 触摸主按钮 (iOS) | navigate to 01.5 Bill Import | 页面左滑 |
| 点击"稍后再说" | 触摸跳过链接 (iOS) | navigate to 01.7 Dashboard（空状态） | 页面淡出 |
| 展开"其他品牌" | 触摸展开按钮 | 显示通用设置步骤 | 展开动画 |
| 返回 | 安卓返回键 | 无操作（不允许返回注册页，注册已完成） | 无反馈 |
| 滚动 | 手势上滑 | 页面内容滚动 | 原生滚动 |

| 出口 C | 点击"暂时跳过" / "稍后再说" | 01.7 Dashboard (空状态) |
| 返回 | 安卓返回键 | 不允许（注册已完成，不能回注册页） |

**边界情况：**
- **用户反复跳转系统设置又返回：** 每次返回都检测权限状态，授权即跳转，未授权不打扰
- **系统设置页找不到 APP：** APP 的 NotificationListenerService 未正确注册，属于技术 bug，兜底显示"如果列表中没有找到 {app_name}，请重启 APP 后重试"
- **用户在系统设置中授权后又取消：** 以最终返回 APP 时的状态为准
- **设备品牌检测失败：** 显示通用引导步骤，展开"其他品牌"Section
- **Android 13+ 通知权限变化：** 部分 Android 13+ 设备可能需要先授予基础通知权限（POST_NOTIFICATIONS），再开启通知使用权；检测到此情况时引导步骤需增加一步
- **用户已经授权过（重装后再进入）：** 检测到权限已开启，直接跳过本页进入 01.5
- **跳过后在 Dashboard 空状态：** Dashboard 显示引导卡片"开启通知读取，解锁自动记账"链接回本页

## Design Decisions

1. **三段式信任叙事（为什么→怎么保护→怎么开）** — 不是上来就要权限，先回答"为什么"建立合理性，再回答"安全吗"消除恐惧，最后才引导操作。信任建设优先于功能完成
2. **厂商适配的设置引导** — Sue 指出不同厂商配置路径有差异。通过自动检测设备品牌并展示对应截图+步骤，消除"我不知道在哪开"的障碍。MVP 覆盖小米/华为/OPPO/Vivo/三星五大厂商
3. **跳过 -> Dashboard 空状态** — Sue 确认跳过后直接进 Dashboard 而非 CSV 导入。理由：不强迫任何数据导入，让用户自主选择时机。Dashboard 空状态提供回来授权的入口
4. **iOS 快捷指令预告** — Sue 确认 iOS MVP 不支持通知读取，但展示"快捷指令方案即将上线"预告，引导 iOS 用户先用 CSV 导入体验
5. **"消费通知"而非"所有通知"** — 标题和副标题明确限定读取范围是"消费通知"，虽然技术上是 NotificationListenerService（读取所有通知），但 APP 只处理支付类通知，文案传达这一事实
6. **安卓返回键不允许返回** — 注册已完成，返回注册页没有意义。如果用户想退出，可以用"跳过"进入 Dashboard
7. **授权返回后自动跳转** — 检测到权限已开启，显示成功动画 1s 后自动进入下一步，减少一次手动点击
8. **产品名称占位符** — `{app_name}` 待品牌命名确定后全局替换

## Technical Notes

- **NotificationListenerService (Android):** 需要在 AndroidManifest.xml 注册 Service，系统级权限，不能通过 Runtime Permission API 请求，必须引导用户手动开启
- **设备检测：** 使用 `react-native-device-info` 获取 `getManufacturer()` 和 `getModel()`
- **深度链接：** `android.provider.Settings.ACTION_NOTIFICATION_LISTENER_SETTINGS` 直接打开通知使用权设置页
- **权限检测：** APP 从后台回到前台时 (AppState change) 检查 NotificationListenerService 是否已启用
- **厂商截图资产：** 需要为每个适配厂商准备 2-3 张系统设置页截图，Phase 6 资产生成阶段产出
- **iOS 平台检测：** 使用 `Platform.OS` 判断，iOS 显示预告+CSV导入，Android 显示通知权限引导
- **Android 13+：** 需要同时处理 `POST_NOTIFICATIONS` runtime permission 和 `NOTIFICATION_LISTENER` 系统设置
