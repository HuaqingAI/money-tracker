## Overview

**Page Purpose:** 引导大强从支付宝导出 CSV 账单文件并上传到 APP，让 AI 获得历史消费数据进行分类展示。通过分步截图教程消除"不知道怎么导出"的障碍，让一个本质上复杂的操作（跨 APP 导出文件）感觉像是"照着做就行"。

**User Situation:** 大强的状态取决于入口路径：
- **Android 授权后进入：** 通知读取已开启，实时捕获在运行。导入历史账单是锦上添花 -- "看看过去几个月钱都花哪了"。心态轻松，因为产品已经开始工作了。
- **iOS 用户进入：** 快捷指令方案未上线，CSV 导入是目前唯一获取数据的途径。心态更迫切 -- "不导就没数据看"。
- **Dashboard 回来：** 已经用了一段时间，想补充历史数据。心态最主动。

**Success Criteria:**
- 教程完成率（看完教程 + 成功上传）>= 50%
- 上传成功率（选择文件后成功解析）>= 95%
- 平均完成时间 < 3分钟（含切换到支付宝操作时间）
- iOS 用户导入率 >= 70%（这是他们的唯一数据来源）

**Entry Points:**
- 从 01.4 Permission 授权成功后进入（Android）
- 从 01.4 Permission iOS "去导入账单"进入
- 从 01.7 Dashboard 空状态或设置入口进入（后续访问）

**Exit Points:**
- CSV 上传成功 -> 01.6 Import Processing（AI 分类处理中）
- 点击"跳过，先看看" -> 01.7 Dashboard（所有用户可用）

## Content & Language Strategy

**叙事框架因入口而异：**

| 入口 | 语气 | 核心信息 |
|------|------|---------|
| Android 授权后 | 轻松、建议式 | "导入历史数据看看全貌，或者先看看新捕获的消费" |
| iOS / 未授权 | 鼓励、引导式 | "导入账单是体验完整功能的第一步" |
| Dashboard 后续访问 | 简洁、效率式 | "补充历史数据，让消费画像更完整" |

**关键策略：**
- 教程用"做什么"而非"点哪里" -- "打开支付宝，进入账单"比"点击底部第四个Tab"更直觉
- 截图做高亮标注 -- 用户扫一眼就知道要点哪里
- "建议近3个月"给明确推荐 -- 避免用户纠结
- 邮箱提示提前打预防针 -- "几分钟内到达""检查垃圾邮件"

**语言：** 界面简体中文，代码/后端英文

## On-Page Interactions

| 交互 | 触发 | 行为 | 反馈 |
|------|------|------|------|
| 页面加载 | 从前序页面进入 | 检测入口路径，显示对应 Headline variant | 内容淡入 |

| 上传中 | 文件上传过程 | 进度条实时更新 | 进度百分比 |

## File Validation

| Rule | Condition | Error Code | Error Message ZH | Error Message EN |
|------|-----------|------------|-----------------|-----------------|
| 文件类型 | 扩展名必须为 .csv | `ERR_FILE_TYPE` | "请选择 CSV 格式的文件" | "Please select a CSV file" |
| 文件大小 | ≤ 10MB | `ERR_FILE_SIZE` | "文件过大，建议选择近3个月的账单" | "File too large — try last 3 months" |
| 文件内容 | 可解析为支付宝账单格式 | `ERR_FILE_PARSE` | "无法识别该文件，请确认是支付宝导出的账单" | "Can't parse this file — please use an Alipay export" |
| 文件编码 | UTF-8 或 GBK | `ERR_FILE_ENCODING` | "文件编码异常，请重新导出" | "File encoding error, please re-export" |

    上传成功│       │跳过(所有用户)
           │       │
           ▼       ▼
    ┌──────────┐  ┌──────────┐
    │ Import   │  │Dashboard │
    │Processing│  │ (01.7)   │
    │ (01.6)   │  │          │
    └──────────┘  └──────────┘

**流转规则：**

| 方向 | 条件 | 目标 |
|------|------|------|
| 进入 A | 01.4 授权成功 (Android) | Bill Import (带跳过选项) |
| 进入 B | 01.4 iOS "去导入账单" | Bill Import (无跳过选项) |
| 进入 C | 01.7 Dashboard 入口 | Bill Import (带跳过选项，如已有通知权限) |
| 出口 A | CSV 上传成功 | 01.6 Import Processing |
| 出口 B | 跳过（所有用户） | 01.7 Dashboard |
| 返回 | 左上角 / 安卓返回键 | 来源页面 |

**边界情况：**
- **选择了非 CSV 文件：** 提示"请选择 CSV 格式的文件"，按钮恢复可点击
- **选择了微信支付的 CSV：** 解析失败，提示"无法识别该文件，请确认是支付宝导出的账单"。未来支持微信后可自动识别来源
- **CSV 文件为空或行数极少（<5笔）：** 正常解析，在 01.6 处理页展示实际结果
- **CSV 文件极大（数千笔/数年数据）：** 正常上传，01.6 处理时间更长但正常工作
- **支付宝更新了导出流程：** 截图教程需要定期更新，标注当前支付宝版本号
- **上传过程中网络中断：** 显示网络错误，保留文件选择状态，用户可一键重试
- **重复导入同一文件：** 后端去重处理，不在前端拦截
- **邮箱中还未收到账单文件：** 用户可能反复回到此页重试，页面状态不做持久化

## Design Decisions

1. **MVP 仅支付宝** — Sue 确认。支付宝 CSV 格式标准、导出路径清晰。微信支付和银行账单作为"即将支持"展示，暗示产品扩展性
2. **分步截图教程** — Sue 确认方案A。4步教程覆盖从打开支付宝到下载 CSV 的全流程，每步配截图高亮标注。比文字描述更直觉
3. **所有用户均可跳过** — Sue 补充确认。避免堵死"想先看看、手动录几单试试"的用户。Android 授权用户提示"新消费会自动记录"，iOS/未授权用户提示"可以先看看，也可以手动记几笔试试"。Dashboard 和设置中提供再次进入导入的入口
5. **Headline 根据入口路径变化** — Android 授权后是"看看全貌"（锦上添花），iOS 是"开始体验"（必要步骤），不同语气匹配不同心态
6. **"建议近3个月"明确推荐** — 避免大强纠结"选多久的？"的决策疲劳
7. **上传成功自动跳转** — 减少手动点击，1.5s 延迟让用户看到"上传成功"反馈
8. **产品名称占位符** — `{app_name}` 待品牌命名确定后全局替换

## Technical Notes

- **文件选择器：** 使用 `react-native-document-picker` 或 Expo `DocumentPicker`，过滤 MIME type `text/csv`
- **CSV 解析：** 支付宝账单 CSV 使用 GBK 编码（部分版本 UTF-8），需要做编码检测和转换
- **支付宝 CSV 格式：** 标准列包含交易时间、交易分类、交易对方、商品说明、收/支、金额、收/付款方式、交易状态。解析时以列头匹配而非列序，兼容格式微调
- **文件大小限制：** 10MB，约可覆盖数万笔交易
- **上传方式：** multipart/form-data 上传到后端，后端解析后入库
- **去重逻辑：** 后端根据交易时间+金额+交易对方做去重，防止重复导入
- **截图资产：** 4张支付宝操作截图需要在 Phase 6 资产生成阶段产出，标注当前支付宝版本号
