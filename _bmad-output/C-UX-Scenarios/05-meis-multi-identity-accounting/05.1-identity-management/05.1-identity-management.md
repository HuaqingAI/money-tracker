## Overview

**Page Purpose:** 创建和管理财务身份（"打工人小美" vs "老板小美"），为每个身份设置交易自动归属规则，让同一张银行卡里的消费被 AI 分割到正确的身份名下。

**核心交互哲学：AI 识别模式，用户确认身份。**

与 02.3 受益人标注的"规则确认"模型类似，但维度不同：02.3 解决"为谁花的"（家庭成员），05.1 解决"以哪个身份花的"（财务角色）。小美的副业消费散布在各个平台（小红书投流、快递寄样品、1688进货），AI 通过商户、金额、时段模式归纳出"这些属于副业"的规则，小美只需确认。

**User Situation:** 小美，28岁设计师+小红书带货。月底午休时间打开APP，想创建副业身份，把副业收支从个人消费中分离出来。她已经用APP几个月了（免费层），积累了足够的交易数据。刚升级付费（解锁多身份功能），心态是"终于可以算清楚了"。

**Success Criteria:**
- 身份创建完成率 >= 90%（进入页面后成功创建身份的用户比例）
- 规则确认完成率 >= 80%（至少确认过一条规则的用户比例）
- AI 规则接受率 >= 70%（用户直接确认 AI 规则的比例）
- 平均完成时间 < 60s（创建身份 + 确认规则）
- 规则覆盖率 >= 75%（AI 规则覆盖的副业交易占全部副业交易的比例）

**Entry Points:**
- 从 Dashboard 底部卡片"解锁多身份核算"进入（首次使用，尚未创建身份）
- 从设置 > 身份管理进入（日常使用）
- 从 05.2 Identity P&L "调整归属规则"链接返回（调整场景）

**Exit Points:**
- 创建身份 + 确认规则完成 -> 05.2 Identity P&L（场景主路径）
- 点击已有身份 -> 05.2 Identity P&L（日常查看）
- 返回 -> Dashboard 或 Settings（视入口而定）

## User Mental State

| 维度 | 状态 |
|------|------|
| 触发事件 | 月底想算清楚副业到底赚了还是亏了，刚升级付费解锁多身份功能 |
| 情绪基调 | 期待 + 轻微焦虑（"终于可以算了，但万一是亏的呢？"）|

| 恐惧 | "不会要我手动把每笔消费归类吧？"（M4- 恐惧再次触发）|

## Content & Language Strategy

**核心叙事："两个你，两本账"**

整个页面的情感策略来自小美的身份切换心理：她不只是在管两本账——她在看清两个自己的真实状况。

| 元素 | 叙事策略 | 核心信息 |
|------|---------|---------|
| Empty State 标题 | 身份共鸣 | "你有不同的财务角色吗？"——不是"请创建账本"的工具语言 |
| Step 1 标题 | 轻松命名 | "给你的副业身份起个名字"——像给角色取名，有游戏感 |
| Step 2 标题 | AI 主导 | "AI 分析了你近 3 个月的交易"——强调 AI 做了重活 |
| 确认按钮 | 二元判断 | "确认"/"不是副业"——比 02.3 的四选一更简单 |
| CTA | 指向结果 | "看损益表"而非"完成"——目标导向 |
| Remaining | 请求而非命令 | "帮忙看看是不是副业？"——与 02.3 一致的轻松语气 |

**身份类型对叙事的影响：**
- 副业 -> "副业身份"、"副业支出/收入"、"损益表"
- 自由职业 -> "自由职业身份"、"业务支出/收入"、"收支报表"
- 投资 -> "投资身份"、"投资支出/收入"、"投资损益"

**禁用词：** "会计"、"账户"（太正式）、"分割"（暗示损失）、"必须标注"

**语言：** 界面简体中文，代码/后端英文

## Identity Rule Engine

**OBJECT ID:** `identity-ai-engine`

### Rule Generation API

| Property | Value |
|----------|-------|
| Endpoint | `POST /api/identities/{identity_id}/rules/generate` |
| Request | `{ user_id, identity_type, date_range: "90d" }` |
| Response | `{ rules: [...], unmatched_txns: [...] }` |

### Rule Object Schema

```json
{
  "rule_id": "irule_001",
  "title_zh": "小红书平台消费",
  "title_en": "Xiaohongshu platform expenses",
  "icon": "📱",
  "transaction_type": "expense",
  "confidence": 0.95,
  "match_criteria": {
    "merchants": ["小红书", "薯店", "聚星"],
    "merchant_keywords": ["小红书", "RED"],
    "category": ["购物", "广告"],
    "amount_range": null
  },
  "matched_transactions": [
    { "txn_id": "t101", "merchant": "小红书", "amount": -18000, "date": "2026-04-08" }
  ],
  "stats": { "count": 18, "total_amount": 320000 }
}
```

### Rule Generation Logic (Business Reference)

| 聚类信号 | 生成规则 | 类型 | 置信度 |
|---------|---------|------|--------|
| 商户含"小红书""薯店""聚星" | "小红书平台消费" | 支出 | High (>90%) |
| 商户含快递公司 + 高频 (>10笔/月) + 金额<50/笔 | "快递发货" | 支出 | High |
| 商户含"1688""阿里巴巴" | "1688进货" | 支出 | High |
| 来源为"薯店结算""品牌合作付款" | "带货收入" | 收入 | High |
| 商户含"拼多多""淘宝" + 金额>200 + 无购物分类 | "电商进货" | 支出 | Medium (60-85%) |
| 退款/售后类交易 + 来源匹配已有副业商户 | "售后退款" | 支出 | Medium |
| 商户为美妆品牌 + 金额偏高 + 无个人消费模式 | "样品采购" | 支出 | Low (<60%) -> unmatched |
| 无明确副业信号 | 不生成规则 | -- | None -> unmatched |

**规则排序策略：**
- 收入规则排在列表末尾（先展示支出，让小美看到"原来花了这么多在副业上"）
- 支出按金额总额降序（大头在前）
- Low confidence（< 60%）不进规则，归入 unmatched

### Rule Persistence

| Property | Value |
|----------|-------|
| Purpose | 确认的规则自动应用于未来新交易 |
| Storage | `identity_rules` 表：`rule_id, identity_id, match_criteria (JSONB), transaction_type, is_active, confirmed_at` |
| Auto-apply | 新交易产生 -> 匹配已确认规则 -> 自动归属到对应身份 |
| Feedback | 用户修改自动归属 -> "还有 N 笔类似交易，是否一起修改？" |

**边界情况：**

| 边界 | 处理 |
|------|------|
| 交易数据太少（< 10 笔/3月） | AI 生成规则少（1-2 条），提示"数据越多分析越准，继续使用后 AI 会发现更多模式" |

## Design Decisions

1. **与 02.3 共享交互范式 -- 规则确认而非逐笔标注** -- 小美月均 50+ 笔交易中约一半与副业相关。逐笔归属 = 25 次操作 = 午休时间不够用。规则确认 4-8 次操作 = 60 秒搞定。核心原则与 02.3 一致：用户做决策，AI 做执行
2. **二元判断（是/不是）比多选更简单** -- 02.3 是四选一（为谁花的），05.1 是二元判断（是不是副业）+ 自动识别收支。操作更快，认知负担更低
3. **收入+支出一起确认** -- 不分两步处理收入和支出。AI 同时归纳收入规则和支出规则，小美一次确认完。损益表需要两边的数据才有意义
4. **身份类型影响 AI 策略** -- 选择"副业"时 AI 侧重电商/物流/广告投放模式；选择"投资"时侧重理财/基金/股票交易。类型是 AI 规则引擎的输入参数
5. **Empty State 用身份共鸣而非功能介绍** -- "你有不同的财务角色吗？"比"创建多身份账本"更有情感共鸣。直接触达小美的"双身份"自我认知
6. **卡片净利润数字一眼可见** -- Identity List 的卡片直接展示净利润。小美不需要点进去就能看到"这个月副业赚了还是亏了"。这是整个功能的核心价值
7. **亏损用橙色不用红色** -- 与 03.x 消费列表的设计原则一致。橙色是"关注"而非"错误"，避免审判感。副业亏损是正常的商业波动，不是道德评判
8. **CTA 指向损益表** -- "看损益表"比"完成"更有吸引力。小美的核心目的不是"设置身份"，而是"看到那个数字"
9. **规则一旦确认永久生效** -- 同 02.3，确认的规则自动应用于未来新交易。下个月的新消费自动归属，小美无需重复操作
10. **支持创建多身份** -- 虽然场景以副业为主，但架构支持多身份（副业+投资+自由职业）。Identity List 可承载多张卡片
11. **默认身份不可删除** -- "个人"身份是兜底。未归属到任何自定义身份的交易自动归入个人身份

## Technical Notes

- **身份创建 API：** `POST /api/identities` body: `{ name, description, icon, type, user_id }`
- **规则生成 API：** `POST /api/identities/{identity_id}/rules/generate` body: `{ date_range: "90d" }` 返回 `{ rules, unmatched_txns }`
- **规则确认 API：** `POST /api/identities/{identity_id}/rules/confirm` body: `{ confirmed_rules: [{ rule_id }], excluded_rules: [{ rule_id }] }`
- **逐笔归属 API：** `POST /api/identities/{identity_id}/transactions/assign` body: `{ assignments: [{ txn_id, type: "expense"|"income"|"exclude" }] }`
- **身份概要 API：** `GET /api/identities?user_id={user_id}&month={YYYY-MM}` 返回身份列表 + 本月概要
- **数据模型：** `identities` 表：`id, user_id, name, description, icon, type, is_default, created_at`
- **交易归属：** `transaction_identities` 表：`txn_id, identity_id, source (rule_auto|user_manual|user_rule_confirm), assigned_at`
- **规则持久化：** `identity_rules` 表：`rule_id, identity_id, match_criteria (JSONB), transaction_type (expense|income), is_active, confirmed_at`
- **自动归属流程：** 新交易 -> 匹配 `identity_rules` 中 `is_active = true` -> 命中则写入 `transaction_identities`
- **默认身份兜底：** 未匹配任何规则的交易自动归属到 `is_default = true` 的身份
