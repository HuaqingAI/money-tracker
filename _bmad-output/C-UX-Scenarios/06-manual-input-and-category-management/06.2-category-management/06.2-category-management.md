## Overview

**Page Purpose:** "AI分类的透明化管理" -- 让用户查看AI的分类规则、修正错误分类、创建自定义分类。核心理念是"AI在替你分类，但你随时可以纠正它" -- 给用户对分类系统的掌控感，同时通过准确率统计展示AI的价值（"看，AI已经帮你自动分了 96% 的消费"）。

**User Situation:** 大强刚手动记了一笔"菜市场50块"，AI 把它分到"食品·生鲜"。他觉得分得对，但顺便想看看AI平时分得怎么样 -- 尤其是"美团"有时候是外卖有时候是团购券，AI 到底怎么区分的？他还想把"拼多多"从"购物·综合"改成"购物·日用品"，因为他在拼多多主要买纸巾洗衣液。

**Success Criteria:**
- 分类规则修改成功率 >= 95%（操作简单不出错）
- 修改后AI分类准确率提升（用户修正的规则被学习并应用）
- 页面停留时间 <= 2min（查看+调整不应花太久，这是工具页不是内容页）
- 自定义分类创建率 <= 10%（说明系统默认分类已足够覆盖大多数场景）

**Entry Points:**
- 06.1 Manual Entry 保存后 Toast 中"调整分类"
- 03.2 Transaction Detail 中修改分类后"管理分类规则"入口
- 08.1 Settings 页面"分类管理"入口
- 深度链接

**Exit Points:**
- 返回来源页面（Settings / Transaction Detail / Toast 来源）
- 点击"iOS快捷指令" -> 06.3 iOS Shortcuts Guide（仅 iOS 设备显示）

## User Mental State

| 维度 | 状态 |
|------|------|
| 触发事件 | 刚记了一笔/改了一笔分类，或在设置里好奇"AI怎么分类的" |
| 情绪基调 | 好奇 + 掌控感（"让我看看AI学到了什么"） |
| 希望 | "一眼看到AI的分类规则，哪里不对我直接改" |
| 恐惧 | "这个页面太复杂、选项太多，改了之后不知道影响什么" |
| 内心问题 | "AI 分类准确率到底多少？""我改了'美团'的规则，以前的记录会跟着改吗？" |
| 耐心程度 | 中 -- 有明确目的（改某条规则或看看整体），但如果操作路径长就不想弄了 |

## Content & Language Strategy

**核心叙事：AI 在替你学习**

分类管理不是让用户"管理分类表"，而是让用户看到"AI 在学习你的消费习惯"并允许纠正。准确率数字是信任锚点 -- "96.2% 意味着 AI 基本靠谱，偶尔需要你帮它校准一下"。

| 维度 | 策略 |
|------|------|
| 页面标题 | "分类管理" -- 明确工具型页面定位 |
| 准确率展示 | 大字号正面色调显示百分比。"96.2%" 比"18笔错误"更正向 |
| 规则展示 | "商户名 → 子分类" 的直觉映射格式，无需解释就能看懂 |
| 回溯修改 | "同时修改历史记录(将修改23笔)" -- 透明告知影响范围 |
| 创建分类 | "创建自定义分类" 而非"添加"，暗示系统分类已经很全，这是个性化补充 |

**语气：** 客观、工具型。不花哨，不推销。

**禁用词：** "智能""学习能力"（过度拟人化）

**语言：** 界面简体中文，代码/后端英文

## On-Page Interactions

| 交互 | 触发 | 行为 | 反馈 |
|------|------|------|------|
| 页面加载 | 进入页面 | 加载准确率数据 + 分类规则列表 | 骨架屏 -> 内容淡入 |
| 展开/折叠分类 | 点击一级分类 Header | 切换该分类下规则条目的显示/隐藏 | 展开/折叠动画 |

| 切换回溯选项 | 点击"同时修改历史记录"复选框 | 切换勾选状态 | 复选框动画 |
| 确认修改规则 | 点击"确认修改" | 保存规则，关闭 BottomSheet，列表更新 | Toast "规则已更新" |

**边界情况：**
- **规则冲突：** 同一商户名被用户改为不同分类（如"美团外卖"和"美团"都存在），优先精确匹配。不自动合并，保持用户的修改意图
- **回溯修改的交易已被手动修改过：** 跳过已被手动修改过分类的交易（尊重用户之前的手动操作），仅修改仍为AI自动分类的交易
- **创建与已有分类重名：** 实时校验，重名时 Input 下方显示"该分类已存在"红色提示
- **分类被删除时仍有关联交易：** MVP 不提供删除分类功能。Phase 2 如需删除，需先将关联交易迁移到其他分类
- **规则列表为空（新用户）：** 显示空状态提示，准确率卡用"--"占位
- **规则数量极多（100+商户）：** 列表使用 FlatList 虚拟化渲染，每个分类 Section 默认折叠

## Design Decisions

1. **准确率放在最顶部** -- 这是用户进入分类管理的"第一印象"。96%的准确率是AI价值的直接证据，建立信任后用户更愿意花时间微调剩余4%
2. **"商户 → 子分类"的规则格式** -- 不用表格、不用多层级树。一行一条规则，左边商户右边分类，中间一个箭头，零学习成本
3. **默认勾选回溯修改** -- 用户改规则的意图通常是"以后都这样分"+"以前的也改过来"。默认勾选减少一步操作，同时显示影响笔数让用户安心
4. **不做合并/拆分分类（MVP）** -- 合并和拆分是高级操作，增加页面复杂度但使用率极低。MVP 只做最核心的：查看规则、修改映射、创建自定义分类
5. **自定义分类挂在一级分类下** -- 不允许创建新的一级分类，只允许在现有一级分类下创建子分类。保持分类体系的结构化，避免用户创建过于碎片化的一级分类
6. **不做分类删除（MVP）** -- 删除涉及关联交易迁移，逻辑复杂。MVP 用户如果创建了不需要的分类，可以忽略它（不影响使用）
7. **折叠式分类列表** -- 8-10个一级分类全展开会很长。前3个（按规则数降序）默认展开，其余折叠，用户按需展开
8. **修正数可点击** -- "18笔被修正"作为链接跳转到交易列表（筛选修正过的），帮助用户回顾自己的修正历史

## Technical Notes

- **准确率计算：** accuracy = (total_auto_classified - manually_corrected) / total_auto_classified * 100。仅统计当月数据
- **规则数据：** 从 AI 分类模型的商户-分类映射表读取。每条规则包含 merchant_keyword、matched_category_id、confidence、usage_count
- **回溯修改：** 后端批量更新 SQL，WHERE merchant LIKE '%{keyword}%' AND category_source = 'ai_auto' AND month = current。手动修改过的记录（category_source = 'manual'）跳过
- **分类层级：** 两级结构 -- 一级分类（餐饮、交通、购物...）+ 子分类（饮品、外卖、打车...）。不支持三级或更深层级
- **虚拟化渲染：** 规则列表使用 SectionList + lazy rendering，折叠的 Section 不渲染子项
