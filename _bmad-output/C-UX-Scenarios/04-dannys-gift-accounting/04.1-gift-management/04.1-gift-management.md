## Overview

**Page Purpose:** "人情关系网全景" -- 让大强快速搜索到某个人的人情往来历史，看到收/送的完整时间线和年度统计，为回礼决策提供事实依据。这是 D3+（清楚人情随礼花了多少 11/15）的核心兑现页。

**User Situation:** 大强在微信上收到同事张三的婚礼请柬（下周六），第一反应是"上次张三给了我多少来着？" 他记得去年张三在儿子满月宴上随了份子钱，但记不清是500还是600。打开APP找"人情账"想查个准确数字，好决定这次该随多少。

**Success Criteria:**
- 搜索到联系人历史记录 <= 3s（从输入到看到结果）
- 人情账功能月活用户使用率 >= 30%（差异化壁垒验证指标）
- 从搜索到点击"获取建议"的转化率 >= 50%（用户需要决策辅助）
- 手动录入一条人情记录 <= 30s（录入门槛够低才有数据积累）

**Entry Points:**
- Dashboard "人情账"功能入口卡片（场景路径）
- 底部 Tab "我的" -> 人情账管理（常规路径）
- AI Chat 中提及人情相关话题时的快捷跳转
- 推送通知"下月有 3 场婚礼"点击进入
- 深度链接 / 推送通知

**Exit Points:**
- 点击联系人的"获取回礼建议" -> 04.2 Gift Smart Suggestion
- 点击"即将到来"事件的"去看建议" -> 04.2 Gift Smart Suggestion
- 点击 "+" -> Action Sheet -> "记一笔" -> 录入流程（页内 Bottom Sheet）
- 点击 "+" -> Action Sheet -> "拍照导入礼簿" -> 04.1a Gift Book Import
- 空状态 "拍照导入礼簿" -> 04.1a Gift Book Import
- 底部 Tab 切换到其他页面
- 返回手势 -> Dashboard

## User Mental State

| 维度 | 状态 |
|------|------|
| 触发事件 | 微信收到张三的婚礼请柬，下周六就要去 |
| 情绪基调 | 轻度焦虑 + 目标明确（"我要查清楚再决定"） |
| 希望 | "一查就知道张三上次给了多少，这次我心里就有底了" |
| 恐惧 | "找不到记录怎么办？万一记错了呢？凭感觉随又怕多了少了" |
| 内心问题 | "张三去年随了500还是600？""我们之间还有没有别的往来？""同事之间婚礼一般随多少？" |
| 耐心程度 | 中高 -- 有明确目标（查张三），愿意花时间找，但如果搜索不好用就烦了 |
| 关键信号 | 搜到"张三"并看到"满月宴 收 ¥600"的那一刻，焦虑消解为确定感 |

## Content & Language Strategy

**核心叙事：人情关系网**

人情账的数据结构是"网络型"而非"线性流水" -- 以"人"为核心节点，每个人下挂着时间线。文案策略是**关系叙事** -- 不是"交易记录"，而是"你和张三的往来故事"。

| 维度 | 策略 |
|------|------|
| 金额表达 | 净额用中性色，不用红绿。人情是社交投资，"送出多"不是"亏损" |
| 关系标签 | 用亲近的称呼（"同事""表哥"），不用"分类""标签"等技术词 |
| 时间线措辞 | "收 ¥600""送 ¥800"，简洁直接，不用"您收到了""您赠送了" |
| 空状态 | "一笔一笔都帮你记着" -- 像一个靠谱的管家，不是冰冷的数据库 |
| AI CTA | "AI帮你算" -- 拟人化，降低技术感。不说"AI推荐算法" |
| 事件类型 | 用生活化词语（"婚礼""满月""乔迁"），不用"场景""类别" |

**语气：** 温暖、实用、不卑不亢。像一个靠谱的老朋友帮你记着这些事。

**禁用词：** "债务""欠款""亏损""赚了"（避免将人情商品化）、"应该""必须"（不做道德判断）

**语言：** 界面简体中文，代码/后端英文

**边界情况：**
- **搜索无结果：** 显示"未找到 \"{keyword}\" 相关联系人"，提供"新建联系人并记一笔"的快捷入口
- **同名联系人：** 关系标签 + 最近事件帮助区分。MVP 不做强制去重，允许同名不同人
- **联系人无往来记录（理论上不可能）：** 联系人必须通过录入记录或 OCR 导入创建，不存在空联系人
- **单个联系人往来极多（20+）：** 时间线正常展示全部，按时间倒序，较旧的在下方
- **跨年往来：** 时间线展示全量（不限年份），年度统计默认当年（可切换）
- **录入表单中断：** Bottom Sheet 下拉关闭前弹出确认"放弃录入？"（仅当已有输入时）
- **从 AI Chat 跳转带搜索参数：** 直接填入搜索词并展示结果，用户无需重复输入
- **人情记录与交易记录的关系：** 人情账独立于消费交易。如果用户在 06.1 手动记账时选择"人情"分类，不自动归入人情账（需要额外确认联系人信息）。人情账需主动录入以保证数据质量
- **年度统计中"送出"包含范围：** 仅统计人情账中的记录，不包含消费交易中标记为"人情"分类的记录
- **即将到来事件已过期：** 事件日期过后 3 天自动从"即将到来"移除。如果没有对应的人情记录，推送提醒"张三婚礼已过，要记一笔吗？"
- **即将到来事件与已有人情记录匹配：** 如果事件对应的联系人+事件类型已有当日记录（如通过 04.2 记录），事件自动标记为"已完成"
- **OCR导入后回到04.1：** 联系人列表自动刷新，年度统计更新，新导入的联系人带"新"标记
- **"即将到来"区域为空时：** 整个 Section 隐藏（不显示空标题），页面从年度统计直接到联系人列表

## Design Decisions

1. **以"人"为核心组织数据** -- 人情账的本质是社交关系网络，不是消费流水。用联系人卡片而非交易列表，让大强自然地"找人"而不是"翻记录"。搜索输入的是人名而非金额或日期
2. **搜索栏常驻可见** -- 大强进入人情账的首要动作是"搜张三"。搜索不能藏在图标后面需要点击展开。常驻搜索栏 + 1 字符即触发，让"搜人"的路径最短
3. **净额用中性色** -- D4-（不想在随礼上失礼或过度 10/15）提示我们：人情是社交义务，不是投资回报。用红色表示净支出会暗示"亏了"，制造不必要的心理负担。中性色只呈现事实
4. **快捷金额 Chips** -- 中国人情金额有强烈的文化约定：200/500/600/800/1000/2000 是最常见区间。提供 Chips 减少键盘输入，同时作为"行情参考"降低决策焦虑
5. **事件类型用 Chip 而非下拉** -- 人情场景有限（7+1），Chip 一目了然不用展开下拉找。视觉上也传达出"人情就这几种场景"的认知框架，降低录入的心理门槛
6. **AI CTA 卡片化而非按钮** -- "获取回礼建议"是场景的核心转折。用卡片+上下文文案（"张三婚礼下周六"）而非单一按钮，让 CTA 自带场景感，大强不用思考这个按钮干什么
7. **Bottom Sheet 录入而非独立页面** -- 录入一条人情记录只需 5-6 个字段，不值得独立页面。Bottom Sheet 保持上下文（能看到列表或详情），录入完直接看到更新的数据
8. **关系标签可选** -- 不是所有人都想分类联系人。关系标签帮助区分同名人和提供 AI 建议的参考，但不强制，不增加录入阻力
9. **时间线而非表格** -- 人情往来是故事性的（"去年满月宴张三随了600"），时间线的视觉表达比表格更贴合叙事感。每个节点是一个人情故事片段
10. **人情账独立于消费交易** -- 虽然人情支出也是"花钱"，但人情账有独特的数据结构（联系人维度、事件类型、收/送双向）。与消费流水混在一起会稀释人情账的独立价值，也会让数据统计混乱
11. **⊕ 按钮改为 Action Sheet 多入口** -- 单条录入只是录入方式之一。拍照导入礼簿是批量场景的杀手功能，表格导入覆盖已有电子化数据的用户。Action Sheet 让三种方式并列，不增加页面复杂度
12. **空状态以导入优先而非单条录入** -- 对有几年人情历史的用户（绝大多数），第一次进人情账最想做的是把历史数据一次性导入，不是一条一条录。空状态 CTA 优先级：拍照导入 > 表格导入 > 手动记一笔
13. **"即将到来"Section 条件显示** -- 只有在有未完成的即将到来事件时才显示这个区域。没有待办时页面保持简洁（统计 + 联系人列表），不展示空的"即将到来"标题。事件来源多元（手动添加/AI Chat/推送确认），数据结构独立于人情记录
14. **事件完成后提示记录** -- "即将到来"事件滑动完成时，如果没有对应的人情记录，轻柔提示而非强制。有些事件可能最终没去，有些可能想晚点再记

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | 人情记录是否支持从消费交易自动关联 | 如果用户有一笔"微信转账 800 备注:张三婚礼"的消费记录，是否提供"归入人情账"的快捷操作？可简化录入但增加复杂度 | 🟡 In Discussion |
| 2 | "即将到来"事件的日历视图 vs 列表视图 | 当前设计为列表形式嵌入主页。如果事件较多（如春节前后），是否需要独立日历视图？建议 MVP 先用列表，观察数据后决定 | 🟡 In Discussion |
| 3 | 联系人是否关联手机通讯录 | 从通讯录导入联系人可降低输入成本，但涉及额外权限申请和隐私顾虑 | 🟡 In Discussion |
| 4 | 单条人情记录是否支持"回礼"关联 | 如张三婚礼送800，是否在时间线上标记"这是对满月宴收600的回礼"？增加数据关联但也增加录入复杂度 | 🟡 In Discussion |
| 5 | AI Chat 创建事件的自动化程度 | 用户在 AI Chat 说"张三下周六结婚"，是自动创建事件还是提示确认？自动化程度越高越方便但也越可能误触发 | 🟡 In Discussion |
