## Overview

**Page Purpose:** "一拍全有" -- 拍一张礼簿照片（或从相册选），AI 自动识别姓名和金额，批量导入人情记录。这是人情账的**冷启动杀手功能** -- 把线下手写礼簿转化为结构化数据，解决"几十条记录不可能一条一条录"的核心阻力。

**User Situation:** 大强翻出去年儿子满月宴的礼簿本子（或拍过的照片），想把当时收到的50多条礼金记录导入APP。手写礼簿上每行写着"张三 600""李四 800""王五 500"，字迹有的潦草但基本可辨认。如果逐条手动录入需要30分钟以上，但拍照导入可能只需要2-3分钟。

**Success Criteria:**
- OCR 识别准确率 >= 85%（姓名+金额双匹配）
- 从拍照到确认保存 <= 3 分钟（50条记录量级）
- 用户对识别结果的编辑修正率 <= 30%（大部分记录不需要手动修正）
- 使用 OCR 导入的用户人情账数据量 >= 手动录入用户的 5 倍

**Entry Points:**
- 04.1 Header ⊕ -> Action Sheet -> "拍照导入礼簿"
- 04.1 Empty State "拍照导入礼簿" CTA

**Exit Points:**
- 导入完成 -> 返回 04.1 Gift Management（列表已更新）
- 取消 / 返回 -> 04.1 Gift Management（无变更）

## User Mental State

| 维度 | 状态 |
|------|------|
| 触发事件 | 翻出了去年满月宴的礼簿，或者第一次打开人情账看到导入引导 |
| 情绪基调 | 期待 + 小怀疑（"拍照真能识别手写的？试试看"） |
| 希望 | "拍一张照片就把50条记录全导进去，太省事了" |
| 恐惧 | "万一识别错了一堆，改起来比手动录还麻烦" |
| 内心问题 | "手写字它能认吗？""识别错了能改吗？""导入后能不能删？" |
| 耐心程度 | 中 -- 愿意花几分钟确认和修正，但如果需要改一半以上就放弃了 |
| 关键信号 | 看到识别结果大部分正确（"张三 600""李四 800"都对了），信任感建立 |

## Content & Language Strategy

**核心叙事：一拍全有**

礼簿导入的体验要像"魔法" -- 一张照片变成几十条结构化数据。文案策略是**信任建立** -- 用户对 OCR 天然怀疑（"手写字能认？"），需要在每个步骤中展示能力和透明度。

| 维度 | 策略 |
|------|------|
| 拍照引导 | 实用Tips（"平放、光线均匀"），不说"请确保图像质量"（技术词） |
| 识别过程 | "逐行辨认姓名和金额" -- 拟人化，暗示AI在仔细看。不说"OCR处理中" |
| 置信度展示 | 高置信度不标记（减少噪音），低置信度用橙色温和提醒。不用红色/感叹号 |
| 结果确认 | "请确认识别结果是否正确" -- 把用户放在"审核者"位置，尊重其权威 |
| 事件信息 | "这些记录来自哪次事件？" -- 口语化提问，不说"请选择事件属性" |
| 成功文案 | "52条记录已保存" -- 强调数量，让用户感受到"一拍就省了30分钟" |

**语气：** 自信但谦逊。"我识别了这些，你来确认对不对。"

**禁用词：** "OCR""识别引擎""图像处理"（技术词）、"错误""失败"（尽量用"不确定""需要确认"）

**语言：** 界面简体中文，代码/后端英文

**边界情况：**
- **手写字迹极度潦草：** OCR 置信度低的条目标记为"需要确认"。如果一整页都无法识别，提示"这张照片不太清晰，试试重新拍摄？"并提供重拍按钮
- **礼簿格式不规则：** 有些礼簿是"姓名 金额"，有些是"金额 姓名"，有些用表格线。AI 应兼容多种常见格式。如果格式完全无法解析，降级到手动录入引导
- **同一姓名出现多次：** 保留为独立记录（可能不同的人同名，也可能同一人分多次）。在确认页标注"与已有联系人同名"
- **金额含非数字字符：** 如"伍佰""500元""500.00"，AI 应统一识别为数字。中文大写数字（壹贰叁）需支持
- **照片模糊/光线不足：** 在 Step 1 拍照后如果图片质量低，提示"照片可能不够清晰，建议重拍"但不阻止继续
- **多页礼簿跨页识别：** 每页独立识别后合并结果，在 Step 3 统一确认。不做跨页去重（不同页可能有同名不同人）
- **OCR 超时（> 30s）：** 显示"识别时间较长，请稍候"。超过 60s 提供"取消"选项
- **网络断开：** OCR 需要上传照片到后端。无网络时提示"需要网络连接才能识别"
- **Step 3 编辑后总金额变化：** 实时更新顶部总金额
- **Step 3 全部取消勾选：** "下一步"按钮禁用 + 提示"至少选择一条记录"
- **Step 4 返回 Step 3：** 保留所有编辑状态（勾选、修正的姓名/金额）
- **导入结果中与已有联系人匹配：** 如果已有"张三（同事）"联系人，新导入的"张三"记录自动关联到该联系人。如有歧义（多个张三），Step 3 中标注让用户确认
- **单次导入记录极多（200+）：** Step 3 列表支持滚动加载，顶部固定摘要区。性能上确保流畅

## Design Decisions

1. **多步骤流程而非一步到位** -- 拍照 -> 识别 -> 确认 -> 补充信息 -> 完成，五步看似多但每一步都有明确目的。合并步骤会让单页过于复杂，且用户需要"审核"识别结果的掌控感
2. **支持多页连拍** -- 礼簿通常不止一页。允许连续拍摄多张后一起识别，而非拍一张处理一张，减少等待次数
3. **低置信度温和标记** -- 不用红色/感叹号/错误，用橙色+文字描述（"姓名不确定"）。用户天然对OCR抱有宽容（"手写嘛，认错几个正常"），温和提示让用户愿意手动修正而非觉得"这破功能不行"
4. **高置信度不标记** -- 大部分记录如果都显示"✅已确认"，反而增加视觉噪音。只标记有问题的，让正确的记录"安静地正确"
5. **事件信息统一填写** -- 礼簿一般是同一次事件的记录（如"儿子满月宴"），所以事件类型、日期、方向在 Step 4 统一填写一次，而非每条记录都选。大幅减少重复操作
6. **方向默认"收到"** -- 中国礼簿 95%+ 是自家办事记录收到的份子钱。默认"收到"减少一步操作
7. **允许带问题继续** -- Step 3 有低置信度记录时不阻止进入 Step 4。用户可能想先导入大部分正确的，之后在人情账详情中慢慢修正个别记录
8. **成功页提供"继续导入"** -- 大强可能有多本礼簿（儿子满月、自己婚礼、乔迁）。一键返回 Step 1 继续下一批，不用回到 04.1 再点一次
9. **不做自动去重** -- 不同页/不同事件可能有同名人且金额相同。自动去重可能误删真实数据，宁可保留交给用户判断
10. **Inline Edit 而非弹窗** -- 修正姓名/金额直接在行内编辑（点击即可编辑），不弹出独立编辑框。50条记录可能要改5-10条，弹窗会极大降低修正效率

## Technical Notes

- **OCR 引擎：** 使用 DeepSeek OCR 2（Product Brief 确认票据识别 91%+），需测试手写体识别率
- **中文手写体挑战：** 礼簿手写字迹差异大，需要训练/微调模型。MVP 阶段可考虑限定"印刷体+规整手写"，严重潦草的降级提示
- **图片上传：** 照片上传到后端 OCR 处理。考虑压缩（保证清晰度的前提下减小体积），上传进度可显示
- **结果数据结构：** 每条 OCR 结果包含 {name: string, amount: number, confidence_name: float, confidence_amount: float, bounding_box: rect}
- **金额解析：** 需处理"500""伍佰""500元""500.00""五百"等多种写法
- **批量保存：** Step 4 确认后一次性创建所有人情记录和联系人（事务操作），避免部分成功部分失败
- **联系人匹配：** 导入时用姓名模糊匹配已有联系人（考虑同音字、简繁体），匹配到则关联，否则新建
