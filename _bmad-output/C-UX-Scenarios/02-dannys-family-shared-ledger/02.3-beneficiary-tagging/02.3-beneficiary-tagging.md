## Overview

**Page Purpose:** 为近期消费添加"为谁花的"受益人维度标签（为自己 / 为配偶 / 为孩子 / 为家庭），让家庭账本从"花了多少"升级为"为谁花了多少"。

**核心交互哲学：确认规则，而非逐笔标注。**

AI 分析交易模式后，归纳出 5-8 条消费规则（如"工作日午餐外卖 -> 为自己"），每条规则覆盖 N 笔交易。大强只需确认规则正确性，一次确认 = 批量标注数十笔。只有 AI 无法归类的少量交易才需要逐笔处理。这是"零记账"基因在标注功能上的延伸 -- 用户做决策，AI 做执行。

**User Situation:** 大强刚创建了家庭账本（可能已发送邀请，也可能选了"稍后邀请"），心态处于"会员领了，账本也建了，继续往下走"的惯性中。他对"受益人标注"这个概念可能不熟悉 -- 需要快速理解"为什么要标注"和"怎么标注"。关键是不能让他觉得"又要一笔一笔手动搞"。

**Success Criteria:**
- 规则确认完成率 >= 85%（至少确认过一条规则的用户比例）
- AI 规则接受率 >= 75%（用户直接确认 AI 规则的比例，高于逐笔模式的 60%）
- 平均完成时间 < 30s（确认 5-8 条规则比浏览 50 笔交易快得多）
- 规则覆盖率 >= 80%（AI 规则覆盖的交易占总交易的比例）
- 场景完成率（从 02.1 进入到返回 Dashboard）>= 80%

**Entry Points:**
- 从 02.2 Waiting State "去标注受益人" 进入
- 从 02.2 Invite State "稍后邀请，先设置受益人" 进入
- 从设置/家庭账本管理中进入（日常使用）

**Exit Points:**
- 点击"完成" -> 01.7 Dashboard（家庭账本视图，场景完成）
- 左上角返回 -> 02.2 Shared Family Ledger 或 01.7 Dashboard（视入口而定）

## Interaction Model

### Two-Phase Design

本页面采用**两阶段交互模型**，核心原则：用户确认规则，AI 执行标注。

```
Phase 1: 规则确认（主交互）                Phase 2: 剩余处理（辅助）
┌──────────────────────────┐          ┌──────────────────────────┐
│  AI 归纳 5-8 条消费规则    │          │  AI 无法归类的少量交易     │
│  每条规则覆盖 N 笔交易     │    →     │  用户逐笔选择或跳过       │
│  用户一键确认/修改规则     │          │  通常 < 5 笔             │
│                           │          │                          │
│  操作量：5-8 次点击       │          │  操作量：0-5 次点击       │
│  覆盖率：80%+ 交易       │          │  覆盖率：剩余 20%        │
└──────────────────────────┘          └──────────────────────────┘
```

**为什么不用逐笔标注：**
- 大强有 50 笔月交易，逐笔浏览 + 确认 = 50 次操作 = 违背"零记账"
- 规则确认 = 5-8 次操作 = 30 秒完成 = 延续"零输入"体验
- 规则一旦确认，**未来新交易自动匹配**，无需重复标注

## Content & Language Strategy

**核心叙事："AI 已经搞定了，你确认一下"**

整个页面的信息层级：
1. AI 的工作成果（规则列表）= 主角
2. 用户的操作（确认/修改）= 配角
3. 手动逐笔标注 = 最后的兜底

| 元素 | 叙事策略 | 核心信息 |
|------|---------|---------|
| 页面标题 | AI 主导叙事 | "AI 已识别你的消费模式" -- AI 做了工作，不是你 |
| 副标题 | 明确承诺 | "无需逐笔标注" -- 直击 D3- 恐惧 |
| 规则卡片 | 可读的规则名 | "工作日午餐外卖"而非"分类=餐饮+金额<50+工作日" |
| Phase 2 提示 | 请求而非命令 | "帮忙看看？"而非"请标注以下交易" |
| 完成按钮 | 允许不完美 | 随时可完成，未覆盖的交易不阻塞 |

**语气：** AI 做了重活，用户只需要点头。轻松、快速、无压力。

**禁用词：** "受益人"（太正式）、"逐笔标注"（恰恰是我们要避免的）、"必须"、"请完成"

**语言：** 界面简体中文，代码/后端英文

## AI Rule Engine

**OBJECT ID:** `tag-ai-engine`

### Rule Generation API

| Property | Value |
|----------|-------|
| Endpoint | `POST /api/tags/beneficiary/rules` |
| Request | `{ user_id, date_range: "30d" }` |
| Response | `{ rules: [...], unmatched_txns: [...] }` |

### Rule Object Schema

```json
{
  "rule_id": "rule_001",
  "title_zh": "工作日午餐外卖",
  "title_en": "Weekday lunch delivery",
  "icon": "🍽",
  "suggested_tag": "self",
  "confidence": 0.92,
  "match_criteria": {
    "category": "餐饮",
    "sub_category": ["外卖", "快餐"],
    "amount_range": [0, 5000],
    "time_pattern": "weekday_lunch",
    "merchants": ["美团外卖", "饿了么", "肯德基", "麦当劳"]
  },
  "matched_transactions": [
    { "txn_id": "t001", "merchant": "美团外卖", "amount": 3200, "date": "2026-04-10" },
    ...
  ],
  "stats": { "count": 12, "total_amount": 38400 }
}
```

### Rule Generation Logic (Business Reference)

| 聚类信号 | 生成规则 | 建议标签 | 置信度 |
|---------|---------|---------|--------|
| 分类=餐饮 + 工作日 + 午间时段 + 金额<50 | "工作日午餐外卖" | 为自己 | High (>85%) |
| 分类=交通 + 通勤时段 (7-9点, 17-19点) | "通勤交通" | 为自己 | High |
| 分类=购物/超市 + 金额>80 | "超市采购（>80元）" | 为家庭 | Medium (60-85%) |
| 分类=母婴/育儿 | "母婴育儿用品" | 为孩子 | High |
| 分类=居住 (房贷/水电/物业) | "房贷水电物业" | 为家庭 | High |
| 分类=教育 + 幼儿关键词 | "幼儿教育" | 为孩子 | High |
| 分类=餐饮 + 周末 + 金额>100 | "周末家庭聚餐" | 为家庭 | Medium |
| 分类=美容/个护 + 女性品牌关键词 | "美妆个护" | 为配偶 | Low (<60%) -> 归入 unmatched |
| 分类=购物/综合 + 无明确信号 | 不生成规则 | -- | None -> 归入 unmatched |

**规则合并策略：**
- 同一标签 + 相似分类的交易优先合并为一条规则（如 "物业费"+"水费"+"电费" -> "房贷水电物业"）
- 单笔交易不生成独立规则（至少 2 笔以上才有"规则"的价值）
- Low confidence（< 60%）的交易不进入规则，归入 unmatched 列表（Phase 2 处理）
- 规则数量控制在 3-10 条之间。过少（< 3）说明数据不足；过多（> 10）增加认知负担

### Rule Persistence

| Property | Value |
|----------|-------|
| Purpose | 确认的规则不仅标注当前交易，**还自动应用于未来新交易** |
| Storage | `beneficiary_rules` 表：`rule_id, user_id, match_criteria (JSON), tag_value, confirmed_at` |
| Auto-apply | 新交易产生时，后端自动匹配已确认规则 -> 自动标注受益人标签 |
| Feedback | 用户在交易详情页修改了某笔自动标注 -> 触发规则修正建议（"还有 N 笔类似交易，是否一起修改？"） |
| Note | 这是"零记账"基因的延伸 -- 标注一次，永久生效。用户不需要每月重新标注 |

- **AI 无法生成规则（数据太少或太杂）：** 降级为 Phase 2 逐笔模式。Banner 变为"给每笔消费标注为谁花的"
- **所有交易被规则覆盖（unmatched = 0）：** 不显示"还有 N 笔"提示，直接可完成
- **规则覆盖交易极少（< 30%）：** 仍以规则模式展示已有规则，但底部剩余提示更突出
- **用户展开规则后想把某笔交易从该规则中"摘出"：** MVP 不支持。如需修改个别交易，完成后在 03.2 Transaction Detail 中单笔修改。Open Question
- **规则确认后新增交易：** 已确认规则自动匹配新交易，无需手动操作
- **日常入口（非首次）再次进入：** 展示未标注的新交易 + 新生成的规则（如有）。已标注交易不再展示
- **配偶（小芳）加入后：** 小芳独立进入标注页，看到的是自己的交易和独立的规则分析
- **完成后 Dashboard 变化：** Dashboard 家庭账本视图增加"受益人维度"分类展示（"为孩子花了 ¥2,000"、"为家庭花了 ¥5,000"）

## Design Decisions

1. **规则确认而非逐笔标注 -- 零记账基因的延伸** -- 逐笔标注 50 笔交易 = 50 次操作 = 违背核心承诺。规则确认 5-8 条 = 5-8 次操作 = 30 秒完成。用户做决策（确认规则），AI 做执行（标注交易）。这是整个产品"零输入"哲学在受益人功能上的一致性体现
2. **规则一旦确认，永久生效** -- 确认的规则不仅标注当前交易，还自动应用于未来新交易。大强这个月确认了"工作日午餐 = 为自己"，下个月新的美团外卖自动标注，无需重复操作。真正的"标注一次，永久生效"
3. **两阶段而非一阶段** -- Phase 1 覆盖 80%+ 交易（规则），Phase 2 处理剩余（逐笔）。Phase 2 完全可选，跳过不影响核心体验。大多数用户在 Phase 1 就完成了
4. **显式确认而非默认确认** -- 每条规则需要用户主动点击"确认"或"修改为..."。不用"AI 预选 = 默认确认"模式，原因：(a) 确保用户至少看过每条规则的内容，避免 AI 错误被无感忽略；(b) 确认后卡片收缩，给用户明确的"这条搞定了"反馈；(c) "全部确认"快捷路径兼顾信任 AI 的用户，经二次确认弹窗防止误操作
5. **确认后卡片收缩 = 进度可视化** -- 已确认的规则卡片收缩为紧凑态（高度缩小 50%），未确认的保持完整展示。滚动列表时一眼区分"已搞定"和"待确认"。这比数字进度条更直觉
6. **规则明细默认折叠** -- 展开查看匹配的具体交易是可选操作。大多数用户看到"12 笔 · 共 ¥384"就够了，不需要确认每一笔
7. **Low confidence 不进规则** -- 置信度低的交易（如"丝芙兰 -> 为配偶？"）不勉强生成规则，而是放入 Phase 2 让用户自行判断。规则列表保持高准确率 = 用户信任度高 = 用户更愿意直接点"确认"
8. **规则内例外 = MVP 事后修正** -- 如果一条规则覆盖的 12 笔交易中有 1 笔例外（如某笔美团实际是家庭聚餐），MVP 不支持在规则确认时"摘出"个别交易。用户先确认规则，再去交易详情页单笔修改例外。原因：(a) 规则内例外操作增加交互复杂度，与"快速确认"目标矛盾；(b) 例外是少数情况，事后修正的成本可接受；(c) 后续版本可升级为"规则明细中标记例外"
9. **四个标签保持简单** -- "为自己/为配偶/为孩子/为家庭"，MVP 不提供自定义。后续版本可扩展（"为父母"等）
10. **Phase 2 的"跳过"是一等公民** -- 跳过不是失败，是允许不完美。未标注的交易归为"未分类"，不影响已标注数据的分析价值
11. **规则名用可读语言** -- "工作日午餐外卖"而非"分类=餐饮+金额<50+工作日午间"。让用户一眼理解规则含义，建立对 AI 分析能力的信任
12. **颜色区分标签** -- 四种颜色在规则卡片中保持一致，为 Dashboard 受益人维度饼图奠定视觉基础
13. **撤销是安全网** -- 每条已确认规则都可撤销恢复为 Pending。操作可逆 = 降低决策压力 = 用户更果断地点"确认"

## Technical Notes

- **规则生成 API：** `POST /api/tags/beneficiary/rules` body: `{ user_id, date_range }` 返回 `{ rules: [...], unmatched_txns: [...] }`。AI 分析在后端完成，前端只渲染结果
- **规则保存 API：** `POST /api/tags/beneficiary/rules/confirm` body: `{ confirmed_rules: [{ rule_id, tag_value }] }` 批量保存规则确认
- **逐笔保存 API（Phase 2）：** `POST /api/tags/beneficiary/batch` body: `{ tags: [{ txn_id, tag }] }`
- **规则引擎：** 后端使用聚类算法（基于分类 + 商户 + 金额 + 时段）将交易分组，每组生成可读规则标题。初期可用规则模板匹配（参考 Rule Generation Logic 表），后续升级为 ML 聚类
- **规则持久化模型：** `beneficiary_rules` 表：`rule_id, user_id, match_criteria (JSONB), tag_value, source (ai|user), is_active, confirmed_at, created_at`
- **自动标注流程：** 新交易 -> 匹配 `beneficiary_rules` 中 `is_active = true` 的规则 -> 命中则自动写入 `transaction_tags` (source=rule_auto)
- **标签数据模型：** `transaction_tags` 表增加 `source` 字段：`ai_rule`（规则自动标注）/ `user_manual`（用户手动标注）/ `user_rule_confirm`（用户确认规则后批量标注）
- **规则反馈循环：** 用户修改自动标注 -> 触发规则修正建议 -> 用户确认后更新规则 -> 未来交易按新规则标注
- **Dashboard 受益人视图：** `GET /api/dashboard/monthly?ledger={family_id}&group_by=beneficiary`
- **离线支持：** 规则确认缓存在本地，网络恢复后同步
- **规则数量限制：** 前端最多展示 10 条规则。超出部分按置信度排序截断，截断的交易归入 unmatched
